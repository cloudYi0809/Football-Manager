# 12. 数据导入与清洗规范

---

# 一、总原则

Android App 不负责抓取数据。数据处理在电脑端完成。

流程：

```text
原始数据 CSV/Excel/JSON
→ Python 清洗
→ ID 映射
→ 字段标准化
→ 数据校验
→ 生成 history.db
→ Android assets 内置
→ 首次启动复制到本地
```

---

# 二、数据目录结构

```text
/data
├── raw/ 原始数据
├── cleaned/ 清洗后 CSV
├── mapping/ ID 映射表
├── output/ SQLite 输出
├── logs/ 清洗日志
└── patches/ 补丁数据
```

# 四、字段标准化

## 1. 日期

统一：YYYY-MM-DD。

## 2. 国籍

统一英文国家名或 ISO code。建议使用 country_code 字段保留 ISO。

## 3. 位置

统一枚举：

- GK
- RB
- LB
- CB
- DM
- CM
- AM
- RW
- LW
- ST

## 4. 惯用脚

- Left
- Right
- Both

## 5. 金额

统一整数，单位欧元或英镑，项目配置中指定。

# 五、ID 映射

由于不同数据源球员 ID 不同，必须建立内部 ID。

## 1. player_id 生成规则

优先使用人工映射。若无映射，则根据：

- 姓名
- 出生日期
- 国籍
- 位置

匹配生成。

## 2. club_id 生成规则

根据：

- 俱乐部名称
- 国家
- 城市
- 创建年份

进行映射。

## 3. 去重规则

疑似重复条件：

- 姓名相同 + 出生日期相同
- 姓名相似 + 国籍相同 + 出生年份相同
- 数据源 ID 不同但履历相同

重复数据需要输出到 duplicate_report.csv。

# 六、能力值映射

能力值统一 1-100。

如果原始数据来自不同评分体系：

## 1. FIFA 评分 1-99

可直接映射。

## 2. FM CA/PA 1-200

映射：

```text
能力 = round(raw / 2)
```

## 3. 统计数据映射

当没有能力值时，可根据真实表现估算：

- 出场数
- 进球
- 助攻
- 联赛强度
- 国家队出场
- 年龄
- 身价
- 荣誉

# 七、历史新星处理

每个历史新星需要配置：

- player_id
- discoverable_from
- initial_region_code
- default_youth_club_id
- default_first_team_club_id
- default_breakthrough_year
- default_transfer_path
- legend_level
- tags

如果开局年份晚于 discoverable_from，则球员应按历史状态出现在对应俱乐部。

例如：

- 1992 开局：罗纳尔多隐藏在巴西可发现。
- 2002 开局：罗纳尔多已在 Real Madrid 或对应俱乐部。

---

# 八、转会数据处理

转会记录需要：

- player_id
- from_club_id
- to_club_id
- transfer_date
- fee
- transfer_type
- is_historical

如果 fee 缺失，填 0，并设置 fee_unknown = 1，后期可加字段。

---

# 九、比赛数据处理

match 表需要：

- season_id
- competition_id
- match_date
- home_club_id
- away_club_id
- home_score_real
- away_score_real

如果真实赛程缺失，可由赛程生成器补齐。

---

# 十、数据校验

导入前必须校验：

1. player_id 不为空。
2. club_id 不为空。
3. 球员出生日期合法。
4. 转会日期合法。
5. 阵容关系 season_id 合法。
6. match 主客队不同。
7. 历史新星 player_id 必须存在。
8. 所有外键可解析。
9. 同一存档不重复生成同一历史新星。

# 十四、验收标准

1. 能从 CSV 生成 SQLite。
2. 能导入 3000+ 球员。
3. 能导入 100+ 俱乐部。
4. 能导入阵容和转会。
5. 能检查重复球员。
6. 能输出错误报告。
7. Android 可读取生成的 history.db。
