# 04. 数据库设计与字段说明

本文档定义 SQLite 数据库逻辑结构。实际开发可以用 Room Entity 映射。

---

# 一、数据库分层

项目建议使用两类数据库：

## 1. history.db

只读历史数据库，随游戏安装包或数据包提供。

包含：

- 球员基础资料
- 俱乐部资料
- 赛季资料
- 历史阵容
- 历史转会
- 历史比赛
- 赛事结构
- 历史新星池
- 历史事件
- 基础球探、教练、经纪人模板

## 2. save_xxx.db

每个存档一个数据库。

包含：

- 当前日期
- 当前世界状态
- 球员当前状态
- 当前阵容关系
- 当前合同
- 当前转会
- 当前伤病
- 当前士气
- 当前任务
- 当前新闻
- 玩家设置

---

# 二、核心实体关系

## 1. 球员关系

player 是基础表。  
player_attributes 提供球员在不同赛季的属性。  
save_player_state 记录存档中的当前状态。

关系：

- player 1:N player_attributes
- player 1:N transfer_history
- player 1:N squad_membership
- player 1:1 save_player_state per save

## 2. 俱乐部关系

club 是基础表。  
club_season_state 记录俱乐部赛季状态。  
youth_academy 记录青训设施。  
squad_membership 记录历史阵容关系。  
save_club_state 记录存档中的当前俱乐部状态。

## 3. 赛事关系

competition 定义赛事。  
competition_season 定义某赛季赛事配置。  
match 定义比赛。  
league_table 记录积分榜。

## 4. 存档关系

save_world_state 是存档主表。  
所有 save_ 前缀表都通过 save_id 关联。

# 三、字段设计原则

## 1. ID 设计

所有主表使用 INTEGER PRIMARY KEY。  
历史数据 ID 与存档状态 ID 分离。  
外部数据源 ID 单独保存 source_id，不作为内部主键。

## 2. 时间字段

统一使用 TEXT ISO 日期：YYYY-MM-DD。  
日期推进由游戏内 DateEngine 处理。

## 3. 金额字段

金额统一用整数，单位建议为欧元或英镑。  
例如 120000 表示 120,000。

## 4. 能力字段

能力值建议范围：1-100。

如果以后要兼容 FM 1-200，可增加 raw_ca/raw_pa 字段。

## 5. JSON 字段

SQLite 可使用 TEXT 存储 JSON，用于：

- 默认转会路径
- 事件条件
- 战术配置
- 新闻参数
- 数据包 metadata

## 1. player

关键字段：

- player_id：内部球员 ID
- real_name：真实姓名
- display_name：显示名

- portrait_path：头像路径

# 五、索引建议

必须建立索引：

- player(real_name)
- player(nationality)
- player(primary_position)
- player_attributes(player_id, season_id)
- squad_membership(season_id, club_id)
- transfer_history(player_id)
- transfer_history(transfer_date)
- match(season_id, competition_id)
- match(match_date)
- save_player_state(save_id, player_id)
- scout_report(save_id, player_id)

---

# 六、数据完整性规则

1. player_id 必须全局唯一。
2. club_id 必须全局唯一。
3. 一个存档中，一个球员同一时间只能属于一个 current_club_id，外租需要 loan_club_id 单独记录。
4. 历史转会如果被玩家打断，不能继续强制执行。
5. 历史新星已经被发现或签约后，不能重复生成。
6. 退役球员不能再作为球员被签约，但可以转员工。
7. 存档推进日期不能早于当前日期。
8. 合同到期日必须晚于当前日期，除非为自由球员。
