# 02_数据来源治理与版本管理

## 一、数据治理目标

本项目的真实感由数据质量决定。数据治理需要解决：

1. 数据从哪里来。
2. 谁是权威来源。
3. 冲突如何处理。
4. 如何版本化。
5. 如何回滚。
6. 如何保证导入后不破坏存档。

## 二、数据分层

### 1. Core Schema 层

定义数据库结构，不包含具体足球数据。

特点：

- 随游戏版本升级。
- 需要迁移脚本。
- 不能频繁破坏兼容性。

### 2. Base Data 层

游戏基础数据包。

内容：

- 虚构球员
- 虚构俱乐部
- 虚构联赛
- 默认规则
- 默认赛事

特点：

- 可公开分发。
- 用于可发布版。

### 3. Historical Local Pack 层

本地历史数据包。

内容：

- 真实球员
- 真实俱乐部
- 历史转会
- 历史赛程
- 历史新星池
- 自定义头像、队徽、球衣

特点：

- 用户本地导入。
- 不随安装包公开分发。
- 每个数据包有独立版本号。

### 4. Save State 层

玩家存档。

内容：

- 当前日期
- 当前世界状态
- 球员当前能力、合同、伤病、士气
- 已发生转会
- 历史节点是否被打断
- AI 俱乐部状态

特点：

- 与数据包绑定。
- 需要迁移和修复机制。

## 三、数据包元信息

每个数据包必须包含 manifest.json。

```json
{
  "pack_id": "historical_2002_private_v1",
  "pack_name": "Historical 2002 Local Pack",
  "pack_type": "local_historical",
  "schema_version": 2,
  "data_version": "0.2.0",
  "start_seasons": ["1992_93", "2002_03"],
  "created_at": "2026-07-02",
  "author": "local_user",
  "distribution": "local_only",
  "contains_real_names": true,
  "contains_real_logos": true,
  "contains_real_faces": false,
  "license_note": "Local personal use only",
  "checksum": "sha256_placeholder"
}
```

## 四、导入数据校验

导入时按以下顺序校验：

1. manifest.json 是否存在。
2. schema_version 是否兼容。
3. CSV/SQLite 表结构是否完整。
4. 主键是否重复。
5. 外键是否可解析。
6. 必填字段是否缺失。
7. 数值范围是否合法。
8. 图片路径是否存在。
9. 起始赛季是否完整。
10. 球员-俱乐部-赛季关系是否闭合。

## 五、数据冲突优先级

当多个来源数据冲突时，按以下优先级：

1. 用户手动修正。
2. 当前数据包内显式覆盖文件。
3. 权威历史数据源。
4. 通用公开统计数据源。
5. 自动推断规则。
6. 默认占位值。

## 六、数据质量等级

每条关键数据建议记录 confidence_level。

| 等级 | 含义 | 用途 |
|---|---|---|
| A | 高置信，已人工确认 | 核心球员、核心俱乐部 |
| B | 多来源一致 | 常规球员 |
| C | 单来源可信 | 普通球员 |
| D | 自动推断 | 补齐字段 |
| E | 占位数据 | 后续需要修正 |

## 七、数据导入流水线

推荐电脑端处理流程：

```text
raw_csv / raw_json
→ normalize_names.py
→ map_clubs.py
→ map_players.py
→ build_season_rosters.py
→ infer_attributes.py
→ validate_integrity.py
→ generate_history_db.py
→ export_pack.zip
```

必须提供：

## 九、数据版本升级策略

### 1. 小版本升级

例如 0.2.0 → 0.2.1。

允许：

- 修正球员字段。
- 修正俱乐部字段。
- 补充头像路径。
- 修正转会费。

不需要破坏存档。

### 2. 中版本升级

例如 0.2 → 0.3。

可能新增表或字段。

需要：

- 提供 migration.sql。
- 存档进入前进行备份。
- 失败时回滚。

### 3. 大版本升级

例如 1.x → 2.x。

可能不兼容旧存档。

需要：

- 明确提示。
- 提供只读打开旧存档。
- 尽量支持导出旧存档摘要。

## 十、数据问题反馈机制

即使是个人版，也建议做“本地问题标注”。

玩家可以在球员详情页标记：

- 出生日期错误
- 位置错误
- 俱乐部错误
- 能力不合理
- 潜力不合理
- 转会路径错误
- 图片缺失

标记保存到 local_data_issues 表，后续通过编辑器批量修正。
