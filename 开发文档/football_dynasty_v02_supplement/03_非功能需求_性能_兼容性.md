# 03_非功能需求、性能与兼容性

## 一、目标设备

### 最低配置

- Android 10 及以上。
- 4GB RAM。
- 可用存储 2GB。
- 中端 CPU 可流畅运行。

### 推荐配置

- Android 12 及以上。
- 6GB RAM 及以上。
- 可用存储 5GB。
- 中高端 CPU。

## 二、应用体积目标

### 可发布基础版

- APK/AAB 基础包：≤ 300MB。
- 内置虚构数据库：≤ 200MB。
- 内置原创图像资源：≤ 300MB。

### 个人本地历史包

- 数据库包：100MB - 800MB。
- 图片资源包：可独立存储，不进入安装包。
- 用户自行控制本地资源大小。

## 三、核心性能指标

| 场景 | 指标 |
|---|---:|
| 冷启动进入主菜单 | ≤ 5 秒 |
| 首次复制基础数据库 | ≤ 60 秒，必须显示进度 |
| 进入阵容页 | ≤ 1 秒 |
| 搜索球员 5000 人 | ≤ 1 秒 |
| 单场比赛快速模拟 | ≤ 1 秒 |
| 单场比赛事件模式 | 实时播放不卡顿 |
| 点击继续一天 | 平均 ≤ 1 秒，峰值 ≤ 3 秒 |
| 转会窗口每日推进 | 平均 ≤ 2 秒，峰值 ≤ 5 秒 |
| 赛季结束结算 | ≤ 15 秒，必须显示进度 |
| 20 年自动模拟 | 不崩溃，可后台运行并显示进度 |

## 四、存档体积指标

| 存档阶段 | 目标体积 |
|---|---:|
| 新开档 | ≤ 50MB |
| 1 个赛季后 | ≤ 100MB |
| 10 个赛季后 | ≤ 500MB |
| 30 个赛季后 | ≤ 1.5GB |

若启用完整比赛事件归档，需允许用户选择“详细历史保留年限”。

## 五、历史归档策略

### 1. 保留完整数据

最近 2 个赛季保留完整数据：

- 每场比赛事件
- 每名球员训练变化
- 每日新闻
- 转会谈判日志

### 2. 压缩归档

2 年前的数据压缩为赛季摘要：

- 比赛结果
- 积分榜
- 球员赛季统计
- 转会记录
- 奖项
- 重大事件

### 3. 删除低价值日志

超过设定年限的低价值日志可删除：

- 普通训练日志
- 普通媒体新闻
- 无影响球探报告
- 已过期谈判失败记录

## 六、数据库索引要求

必须建立索引：

- player.real_name
- player.normalized_name
- player.birth_date
- player.nationality
- player.primary_position
- squad_membership.club_id + season_id
- transfer_history.player_id
- transfer_history.transfer_date
- match.season_id + competition_id
- match.match_date
- save_player_state.save_id + player_id
- scout_report.save_id + player_id

## 七、异步任务要求

以下任务必须异步执行，并显示进度：

- 数据包导入
- 数据库迁移
- 赛季结束结算
- 长程自动模拟
- 批量球员搜索索引构建
- 大量头像路径校验
- 存档压缩

## 八、崩溃保护

以下操作前必须自动备份：

- 数据库版本迁移
- 赛季结算
- 导入新数据包
- 删除数据包
- 存档压缩
- 大版本更新首次打开旧存档

## 九、电量与后台限制

长程模拟时：

- 默认保持屏幕常亮提示。
- 支持暂停。
- 支持中断后从最近 checkpoint 恢复。
- 每模拟 30 天写入一次轻量 checkpoint。
- 每模拟 1 个赛季写入一次完整 checkpoint。

## 十、可观测性日志

开发版必须记录：

- 每日推进耗时。
- AI 转会决策耗时。
- 比赛模拟耗时。
- 数据库查询慢 SQL。
- 存档大小变化。
- 每赛季转会费总额。
- 每赛季平均工资。
- 每赛季进球分布。
- 每赛季伤病数量。

日志写入 perf_log 表，便于调参。
