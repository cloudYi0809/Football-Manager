# 09_长程性能、存档与版本迁移

## 一、设计目标

足球经理类游戏最重要的资产是存档。必须保证：

1. 存档不轻易损坏。
2. 升级后尽量兼容。
3. 长程模拟体积可控。
4. Android 本地数据库查询稳定。
5. 用户可以备份、恢复、复制存档。

## 二、数据库拆分

建议三库结构：

### 1. history.db

只读历史基础库。

内容：

- 球员基础资料
- 俱乐部基础资料
- 初始赛季数据
- 历史转会
- 历史新星池
- 赛事结构

### 2. save_xxx.db

玩家存档库。

内容：

- 当前世界状态
- 当前球员状态
- 当前俱乐部状态
- 已发生转会
- 伤病
- 新闻
- 赛季统计
- 被打断历史节点

### 3. cache.db

可重建缓存库。

内容：

- 搜索索引
- 排名缓存
- 球员筛选缓存
- 图片路径缓存
- 统计榜单缓存

cache.db 可删除重建。

## 三、存档版本号

每个存档必须记录：

```json
{
  "save_id": "save_001",
  "game_version": "0.2.0",
  "schema_version": 2,
  "data_pack_id": "historical_2002_private_v1",
  "data_pack_version": "0.2.0",
  "created_at": "2026-07-02",
  "last_played_at": "2026-07-02",
  "current_date": "2003-05-30"
}
```

## 四、迁移策略

### 1. 自动迁移

适用于：

- 新增 nullable 字段。
- 新增表。
- 新增索引。
- 新增缓存字段。

### 2. 手动迁移

适用于：

- 字段拆分。
- 表结构重构。
- 枚举值改变。
- 核心关系改变。

### 3. 不兼容迁移

适用于：

- 存档底层数据包变化过大。
- 关键历史数据 ID 改变。
- 核心模拟规则无法转换。

处理：

- 提示无法直接升级。
- 提供只读打开。
- 提供导出球队历史摘要。

## 五、迁移流程

1. 检测存档 schema_version。
2. 创建完整备份。
3. 校验剩余存储空间。
4. 执行迁移脚本。
5. 校验关键表完整性。
6. 失败回滚。
7. 成功后更新版本号。

## 六、存档 checkpoint

长程模拟时创建 checkpoint。

| 类型 | 频率 | 内容 |
|---|---|---|
| light_checkpoint | 每 30 天 | 日期、关键状态、队伍表 |
| season_checkpoint | 每赛季 | 完整赛季结束状态 |
| migration_backup | 迁移前 | 完整备份 |
| user_backup | 用户手动 | 完整备份 |

## 七、赛季归档

赛季结束后执行归档：

1. 生成赛季摘要。
2. 压缩比赛事件。
3. 删除低价值每日日志。
4. 聚合球员赛季统计。
5. 聚合俱乐部财务统计。
6. 生成历史榜单。
7. vacuum 数据库。

## 八、存档损坏检测

打开存档时检查：

- manifest 是否存在。
- 关键表是否存在。
- save_world_state 是否只有一条当前状态。
- 当前执教俱乐部是否存在。
- 当前日期是否合法。
- 球员状态外键是否完整。
- 比赛表是否有孤儿记录。

## 九、存档修复策略

轻微错误：

- 重建缓存。
- 删除孤儿日志。
- 回退缺失头像。
- 修复索引。

中等错误：

- 使用最近 checkpoint 恢复。
- 重建积分榜。
- 重建统计榜。

严重错误：

- 提示用户恢复备份。
- 导出可救援数据。

## 十、性能优化策略

1. UI 只查询分页数据。
2. 球员搜索使用 normalized_name + FTS。
3. 每日推进只处理活跃范围。
4. 非玩家关注联赛可使用简化模拟。
5. 旧赛季只保留摘要。
6. AI 决策按频率执行，不每日全量计算。
7. 图片资源懒加载。
8. 缓存榜单和统计。

## 十一、活跃范围机制

玩家关注范围：

- 玩家所在联赛：完整模拟。
- 玩家参与杯赛：完整模拟。
- 主要欧战：完整模拟。
- 关注球员所在队：中等模拟。
- 其他联赛：简化模拟。

简化模拟只生成：

- 比分
- 进球者
- 红黄牌
- 伤病
- 关键评分

不生成完整事件流。
